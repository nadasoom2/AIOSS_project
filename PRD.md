## 1. 개요 및 배경

### 1.1 문제 정의

동아대학교에 재학 중인 외국인 유학생들은 비자 연장, 외국인등록증 발급, 시간제취업허가 등 행정 업무를 처리할 때 다음과 같은 어려움을 겪는다.

- **언어 장벽:** 전체 유학생들중 TOPIK 3급(간단한 의사소통 수준)이상 취득 비율이 30%정도로 낮음
                -> 한국어로 작성된 행정/절차를 이해하기 어렵다
- **정보 분산:** 관련 정보가 학교 홈페이지, 출입국관리사무소 등 여러 채널에 흩어져 있다.

### 1.2 제품 개요

위 문제를 해결하기 위해, 유학생의 행정·생활 관련 질문에 **한국어·영어·중국어**로 즉각 응답하는 AI 에이전트를 구축한다. 이 에이전트는 단순 FAQ 봇이 아니라, 질문의 의도와 엔티티를 파악한 뒤 최적의 경로(Fast / Hybrid / Slow)로 처리하는 **파이프라인 기반 RAG 시스템**이다.

### 1.3 핵심 설계 원칙

- **하드코딩 금지:** 키워드, 임계값, 프롬프트 등 변경 가능한 모든 값은 DB에 저장한다.
- **비용 최적화:** 반복 질문은 LLM 호출 없이 템플릿으로 처리한다.
- **다학교 확장 가능성:** 학교(scope) 단위로 데이터를 분리하여 타 학교로 확장이 가능하도록 설계한다.

## 2. 목표 및 성공 지표

### 2.1 비즈니스 목표

| 목표 | 세부 내용 |
|------|----------|
| 담당자 반복 업무 경감 | 국제처 직원이 응대하는 반복 질문 수 30% 이상 감소 |
| 유학생 정보 접근성 향상 | 행정 처리 완료까지 소요 시간 단축 |
| 다국어 지원 | 한·영·중 3개 언어로 동등한 품질의 답변 제공 |

### 2.2 성공 지표 (KPI)

| 지표 | 목표값 | 측정 방법 |
|------|--------|----------|
| Fast Path 비율 | 운영 3개월 후 ≥ 60% | `query_log` 통계 |
| 평균 응답 시간 | ≤ 1.5초 (전체 평균) | `query_log.response_time` |
| Fast Path 응답 시간 | ≤ 0.2초 | `query_log.response_time` WHERE route = 'fast' |
| Slow Path 응답 시간 | ≤ 3초 | `query_log.response_time` WHERE route = 'slow' |
| Intent 분류 정확도 | ≥ 90% | 주기적 샘플링 검수 |
| 사용자 만족도 | 긍정 피드백 ≥ 75% | 인터페이스 피드백 버튼 |

## 3. 사용자 페르소나

### Persona A — 입학 초기 유학생 (신규)
- **배경:** 한국 입국 후 1~2주 이내. 한국어 낮음 (영어 또는 중국어 사용).
- **Pain Point:** 외국인등록증, 건강보험 가입, 학생증 발급 등 처음 해야 할 일을 모름.
- **주요 질문 유형:** `ACTION_GUIDE` — "한국 온 지 이틀인데 뭐부터 해야 해?"

### Persona B — 재학 중 유학생 (일반)
- **배경:** 학기 중 재학. 비자 만료 예정 또는 아르바이트 시작 예정.
- **Pain Point:** 비자 연장 절차, 시간제취업허가 조건을 매번 새로 찾아야 함.
- **주요 질문 유형:** `PROCEDURE`, `CONDITION_CHECK`, `REQUIREMENT`

### Persona C — 문제 상황 유학생 (긴급)
- **배경:** 비자 초과 체류, 서류 분실 등 문제가 발생한 상황.
- **Pain Point:** 어디에 연락해야 하는지, 불이익이 얼마인지 알 수 없어 불안함.
- **주요 질문 유형:** `PROBLEM_SOLVE`

## 4. 사용자 스토리

### 4.1 재학 중

> **As** 한국에서 학교생활을 하던 유학생으로,  
> **I want** 학기 중 시점에 맞춰 먼저 처리해야 할 행정 업무 목록과 순서를 알고 싶다,  
> **So that** 수업·아르바이트와 병행하면서도 중요한 절차를 놓치지 않고 제때 처리할 수 있다.

**수용 기준:**
- 학기 일정(수강, 시험 기간 등)을 고려해 지금 시점에 필요한 업무를 우선순위로 안내한다.
- 각 업무의 신청 기한, 준비 서류, 미이행 시 불이익을 함께 명시한다.
- 영어 또는 중국어로 질문해도 동일한 구조와 핵심 정보로 답변을 제공한다.

---

> **As** D-4 비자를 가진 유학생으로,  
> **I want** 아르바이트 가능 여부와 조건을 알고 싶다,  
> **So that** 합법적으로 시간제 취업을 시작할 수 있다.

**수용 기준:**
- D-4 비자 기준의 취업 가능 조건(6개월 경과, TOPIK 요건 등)을 정확히 안내한다.
- 가능/불가능 여부를 먼저 명시한 후 근거를 설명한다.
- 신청 절차와 필요 서류까지 함께 제공한다.

## 5. 기능 요구사항

### 5.1 다국어 엔티티 추출 (FR-01)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-01-1 | 한국어·영어·중국어 질문에서 핵심 엔티티(비자 유형, 지역, 행정 업무, 서류명)를 추출해야 한다. | P0 |
| FR-01-2 | 엔티티 매핑은 `variable_dictionary` 테이블을 참조하며, 동의어(알바/아르바이트/part-time job/打工)를 표준 용어로 통합한다. | P0 |
| FR-01-3 | 신규 용어는 DB 행 추가만으로 코드 수정 없이 반영된다. | P0 |
| FR-01-4 | 추출된 엔티티를 변수 슬롯(`{LOCATION}`, `{TASK}`, `{VISA_TYPE}`)으로 추상화한 패턴 문자열을 생성한다. | P1 |

### 5.2 하이브리드 Intent 분류 (FR-02)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-02-1 | 6가지 Intent(`PROCEDURE`, `CONDITION_CHECK`, `REQUIREMENT`, `INFO_SEARCH`, `ACTION_GUIDE`, `PROBLEM_SOLVE`)를 분류한다. | P0 |
| FR-02-2 | 규칙 기반(Phase 1)과 임베딩 기반(Phase 2)을 조합한 하이브리드 분류기를 사용한다. | P0 |
| FR-02-3 | Phase 1에서 단일 Intent 매칭 시 (confidence = high) Phase 2를 건너뛴다. 전체 요청의 60~70%가 Phase 1에서 종료되어야 한다. | P1 |
| FR-02-4 | Phase 2에서 생성한 질문 임베딩 벡터는 이후 라우팅 및 검색 단계에서 재사용한다 (중복 API 호출 금지). | P1 |
| FR-02-5 | 분류 키워드와 임계값은 DB(`intent_config`, `intent_classifier_config`)에서 관리한다. | P0 |

**Intent 정의표:**

| Intent | 사용자 의도 | 기대 답변 형태 |
|--------|------------|--------------|
| PROCEDURE | 절차·방법을 알고 싶다 | 단계별 리스트 |
| CONDITION_CHECK | 가능 여부를 확인하고 싶다 | 가능/불가능 + 근거 |
| REQUIREMENT | 준비물·기한을 알고 싶다 | 체크리스트 |
| INFO_SEARCH | 일반 정보를 알고 싶다 | 요약 설명 |
| ACTION_GUIDE | 지금 뭘 해야 할지 모르겠다 | 우선순위별 행동 안내 |
| PROBLEM_SOLVE | 문제가 생겼다 | 원인 진단 + 해결 방법 |

### 5.3 경로 라우팅 (FR-03)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-03-1 | Intent 기본 경로 설정(`fast_first` 또는 `slow_direct`)에 따라 1차 라우팅한다. | P0 |
| FR-03-2 | `fast_first` 경로에서 템플릿 유사도 ≥ 0.95이면 Fast Path, 0.85~0.94이면 Hybrid Path, < 0.85이면 Slow Path로 라우팅한다. | P0 |
| FR-03-3 | 유사도 임계값(0.95, 0.85)은 DB에서 관리하며 코드 수정 없이 조정 가능하다. | P1 |
| FR-03-4 | `ACTION_GUIDE`, `PROBLEM_SOLVE`는 `slow_direct`로 설정하여 템플릿 매칭을 건너뛴다. | P0 |

### 5.4 답변 생성 — Fast Path (FR-04)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-04-1 | LLM API를 호출하지 않고 `template_store`의 답변 틀에 엔티티를 대입하여 답변을 조립한다. | P0 |
| FR-04-2 | 답변 틀의 빈 슬롯(`{steps}` 등)은 `knowledge_chunks`에서 DB 조회로 채운다. | P0 |
| FR-04-3 | 목표 응답 시간: ≤ 0.2초 | P0 |

### 5.5 답변 생성 — Hybrid Path (FR-05)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-05-1 | 유사 템플릿을 기반 구조로 활용하고, 차이가 있는 부분만 LLM이 보완한다. | P0 |
| FR-05-2 | `knowledge_chunks`에서 차이 부분에 해당하는 정보를 추가 검색하여 LLM 컨텍스트로 제공한다. | P0 |
| FR-05-3 | 목표 응답 시간: ≤ 1.5초 | P1 |

### 5.6 답변 생성 — Slow Path / RAG (FR-06)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-06-1 | 질문 임베딩 벡터를 사용하여 `knowledge_chunks`에서 관련 문서를 벡터 검색한다. | P0 |
| FR-06-2 | Intent에 따라 검색 전략(우선 태그, 결과 수)을 달리 적용한다 (`search_strategy` 테이블 참조). | P1 |
| FR-06-3 | 비자 유형이 추출된 경우 `visa_info`에서 해당 비자 조건 정보를 함께 컨텍스트에 포함한다. | P0 |
| FR-06-4 | Intent별 시스템 프롬프트(`prompt_template` 테이블)를 사용하여 LLM 답변 구조를 제어한다. | P0 |
| FR-06-5 | 목표 응답 시간: ≤ 3초 | P1 |

### 5.7 비자 유형별 분기 (FR-07)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-07-1 | 동일한 질문이라도 비자 유형(D-2, D-4, D-4-1, D-4-7 등)에 따라 다른 답변을 생성한다. | P0 |
| FR-07-2 | 비자별 조건 데이터는 `visa_info` 테이블에 JSONB로 저장하며, 코드 수정 없이 조건 업데이트가 가능하다. | P0 |

### 5.8 피드백 루프 및 템플릿 자동 등록 (FR-08)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-08-1 | Slow Path에서 생성된 답변이 품질 기준을 통과하면 `template_store`에 자동 등록한다. | P1 |
| FR-08-2 | 등록 대상 Intent는 `PROCEDURE`, `CONDITION_CHECK`, `REQUIREMENT`, `INFO_SEARCH`에 한정한다. | P1 |
| FR-08-3 | 등록 전 중복 검사를 수행하며, 기존 템플릿과 유사도 ≥ 0.93이면 등록하지 않는다. | P1 |
| FR-08-4 | 답변 내 엔티티 값을 변수 슬롯으로 추상화한 후 등록한다. | P1 |
| FR-08-5 | 등록 기준(최소 길이, 불확실 표현 비율 등)은 `feedback_config` 테이블에서 관리한다. | P2 |

### 5.9 로깅 및 모니터링 (FR-09)

| ID | 요구사항 | 우선순위 |
|----|----------|---------|
| FR-09-1 | 모든 질문의 처리 결과를 `query_log` 테이블에 기록한다. (원본 질문, 추상화 질문, Intent, 경로, 유사도, 응답 시간, 답변 전문) | P0 |
| FR-09-2 | Fast / Hybrid / Slow 비율, 평균 응답 시간, 자주 묻는 질문 유형을 조회할 수 있는 통계 API를 제공한다. | P2 |

## 6. 비기능 요구사항

### 6.1 성능

| 항목 | 요구사항 |
|------|----------|
| Fast Path 응답 시간 | p95 ≤ 0.5초 |
| Slow Path 응답 시간 | p95 ≤ 4초 |
| 동시 요청 처리 | 최소 50 RPS 처리 가능 (FastAPI 비동기) |
| 임베딩 API 호출 | 전체 요청의 30~40% 이하 유지 |

### 6.2 정확도

| 항목 | 요구사항 |
|------|----------|
| Intent 분류 정확도 | ≥ 90% (주기적 샘플링 검수 기준) |
| 비자별 분기 정확도 | 비자 유형이 추출된 경우 100% 비자 조건 반영 |
| 오래된 정보 제공 방지 | `knowledge_chunks` 및 `template_store`의 정보 확인일을 답변에 명시 |

### 6.3 확장성

| 항목 | 요구사항 |
|------|----------|
| 다학교 확장 | `scope` 필드로 학교별 데이터를 분리하여, 신규 학교는 데이터 추가만으로 온보딩 가능 |
| 신규 언어 추가 | `variable_dictionary`에 언어 열 추가, `intent_config`에 키워드 추가만으로 지원 |
| 신규 Intent 추가 | DB 행 추가만으로 코드 수정 없이 신규 Intent 정의 가능 |

### 6.4 보안

| 항목 | 요구사항 |
|------|----------|
| 개인정보 미수집 | 사용자 식별 정보를 `query_log`에 저장하지 않는다. |
| API 키 관리 | LLM / 임베딩 API 키는 환경 변수로 관리하며, 코드에 포함하지 않는다. |

### 6.5 유지보수

| 항목 | 요구사항 |
|------|----------|
| 무중단 설정 변경 | 키워드, 임계값, 프롬프트 변경 시 서버 재시작이 불필요하다. |
| 데이터 업데이트 주기 | `knowledge_chunks`는 학기별(연 2회) 이상 정기 검수, 비자 정책 변경 시 즉시 반영 |

## 7. 시스템 아키텍처 요약

```
사용자 질문
    │
    ▼
[1단계] 엔티티 추출 + 하이브리드 Intent 분류
    │   ├─ Phase 1: 규칙 기반 키워드 매칭 (confidence = high → 종료)
    │   └─ Phase 2: 임베딩 유사도 비교 (애매하거나 미매칭 시 실행)
    │
    ▼
[2단계] 경로 라우팅
    │   ├─ fast_first: Template_Store 유사도 검색
    │   │   ├─ ≥ 0.95 → Fast Path
    │   │   ├─ 0.85~0.94 → Hybrid Path
    │   │   └─ < 0.85 → Slow Path
    │   └─ slow_direct (ACTION_GUIDE, PROBLEM_SOLVE) → Slow Path
    │
    ├─── [3A] Fast Path: 템플릿 엔티티 대입 → 답변 조립
    ├─── [3B] Hybrid Path: 템플릿 + LLM 보완
    └─── [3C] Slow Path: RAG 벡터 검색 + LLM 생성
    │
    ▼
[4단계] 최종 답변 출력 + query_log 기록
    │
    ▼
[5단계] 피드백 루프: 품질 검증 → Template_Store 자동 등록
```

## 8. 데이터 요구사항

### 8.1 DB 테이블 목록

#### 설정 테이블

| 테이블 | 역할 |
|--------|------|
| `intent_config` | Intent 종류, 분류 키워드, 라우팅 기본 경로 |
| `intent_examples` | 임베딩 분류용 대표 문장 및 벡터 |
| `intent_classifier_config` | 분류기 임계값 (`embedding_threshold` 등) |
| `search_strategy` | Intent별 벡터 검색 전략 (우선 태그, 결과 수) |
| `prompt_template` | Intent별 LLM 시스템 프롬프트 (버전 관리 포함) |
| `feedback_config` | 템플릿 자동 등록 기준 |

#### 데이터 테이블

| 테이블 | 역할 |
|--------|------|
| `variable_dictionary` | 다국어 엔티티 매핑 사전 (동의어 포함, `scope` 필드) |
| `visa_info` | 비자 유형별 조건과 제한 (JSONB) |
| `knowledge_chunks` | RAG용 지식 저장소 (원문 조각 + 임베딩 벡터) |
| `template_store` | 답변 패턴 캐시 (추상화 패턴 + 엔티티 슬롯 + 벡터) |

#### 운영 테이블

| 테이블 | 역할 |
|--------|------|
| `query_log` | 질문 처리 이력 (Intent, 경로, 응답 시간, 답변 전문) |

### 8.2 `knowledge_chunks` 초기 데이터 요구사항

| 카테고리 | 필수 포함 내용 |
|----------|--------------|
| 비자 관련 | 비자 연장 절차, 비자 유형별 취업 조건, 체류 자격 외 활동 허가 |
| 외국인 등록 | 외국인등록증 발급·재발급·분실 신고 |
| 건강보험 | 가입 방법, 미납 처리, 환급 절차 |
| 시간제 취업 | 신청 절차, 비자별 허용 시간, 위반 시 불이익 |
| 생활 안내 | 부산 교통, 지역별 행정 기관 위치, 은행 계좌 개설 |
| 학사 행정 | 수강 신청, 학적 변경, 장학금 |

### 8.3 `variable_dictionary` 초기 데이터 요구사항

- 지원 언어: 한국어(`ko_val`), 영어(`en_val`), 중국어(`zh_val`)
- 필수 카테고리: `VISA_TYPE`, `LOCATION`, `TASK`, `DOCUMENT`
- 초기 용어 수: 최소 200개 이상 (동의어 포함)

## 9. API 인터페이스 명세

### 9.1 질문 처리 엔드포인트

```
POST /api/v1/query
```

**Request Body:**
```json
{
  "text": "비자 연장 어떻게 해?",
  "language": "ko",
  "visa_type": "D-4",
  "scope": "donga"
}
```

**Response:**
```json
{
  "answer": "비자 연장 방법 안내입니다...",
  "intent": "PROCEDURE",
  "route": "fast",
  "response_time_ms": 180,
  "info_date": "2025-09-01"
}
```

| 필드 | 타입 | 설명 |
|------|------|------|
| `text` | string (필수) | 사용자 질문 원문 |
| `language` | string (선택) | `ko` / `en` / `zh`. 미입력 시 자동 감지 |
| `visa_type` | string (선택) | 알고 있는 경우 선입력. 미입력 시 질문에서 추출 시도 |
| `scope` | string (선택) | 학교 식별자. 기본값: `donga` |

### 9.2 통계 엔드포인트

```
GET /api/v1/stats?scope=donga&from=2026-01-01&to=2026-03-31

**Response:**

{
  "total_queries": 1240,

    "fast": 0.61,
    "hybrid": 0.18,
    "slow": 0.21
  },
  "avg_response_time_ms": {
    "fast": 195,

  },
  "top_intents": [
    { "intent": "PROCEDURE", "count": 450 },

  ]
}
```

## 10. 스코프 정의

### In Scope (v1.0)

- 동아대학교 유학생 대상 비자·행정·생활 안내
- 한국어 / 영어 / 중국어 3개 언어 지원
- 6가지 Intent 분류 및 3경로 처리 파이프라인
- 피드백 루프를 통한 템플릿 자동 축적
- REST API 제공 (프론트엔드 별도)

### Out of Scope (v1.0, 추후 검토)

- 실시간 행정 기관 API 연동 (출입국관리사무소 등)
- 대화 히스토리 기반 멀티턴 대화
- 개인 비자 만료일 알림 서비스
- 타 대학 데이터 동시 서비스 (확장 구조는 설계에 포함)
- 관리자 UI (템플릿·지식 데이터 편집)

**완료 기준:** Intent 분류 정확도 ≥ 85%, 3경로 정상 동작 확인

### Phase 2 — 품질 고도화 (M7~M9)

**완료 기준:** Fast Path 비율 ≥ 40%, 평균 응답 시간 ≤ 2초

- [ ] 타 대학 확장 대비 `scope` 분리 검증

**완료 기준:** Intent 분류 정확도 ≥ 90%, Fast Path 비율 ≥ 60%, 사용자 만족도 ≥ 75%
